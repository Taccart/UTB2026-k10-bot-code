## Overview

The **WiFiService** is a critical infrastructure service that manages all WiFi connectivity for the K10 Bot. It handles both WiFi station mode (connecting to an access point) and access point mode (creating its own network) with automatic fallback functionality.

## Features

### Core Functionality
- **Station Mode**: Connect to an existing WiFi access point
- **Access Point Mode**: Create a WiFi access point when connection fails
- **Automatic Fallback**: Automatically switches to AP mode if station connection fails
- **Connection Management**: Handles WiFi connect/disconnect operations
- **Network Information**: Provides IP address, SSID, and hostname information

### Configuration Persistence
- Settings are saved via `SettingsService` integration
- Automatic loading of previous WiFi credentials on startup
- Persistent hostname and network preferences

## API Methods

### Public Methods

#### `bool wifi_activation()`
Activates the WiFi service with automatic fallback logic:
- Attempts to connect to saved WiFi credentials
- Falls back to AP mode if connection fails
- Returns `true` on successful activation (either mode)

#### `bool registerRoutes(AsyncWebServer *webserver)`
Registers WiFi management HTTP routes (if implemented)
- WiFi status endpoint
- Network information endpoint
- Configuration endpoints

#### `std::string getIP()`
Returns the current IP address:
- Station mode: Local IP on the network
- AP mode: Gateway IP (typically `192.168.4.1`)

#### `std::string getSSID()`
Returns the current SSID:
- Station mode: Connected network name
- AP mode: K10 Bot's access point name

#### `std::string getHostname()`
Returns the configured hostname for the device

#### Service Lifecycle Methods
- `bool initializeService()` - Initializes WiFi hardware
- `bool startService()` - Starts WiFi connectivity with fallback
- `bool stopService()` - Disconnects and stops WiFi
- `bool saveSettings()` - Persists WiFi configuration
- `bool loadSettings()` - Loads saved WiFi configuration

### Protected Methods

#### `bool open_access_point()`
Creates a WiFi access point:
- Default SSID: `K10-Bot-XXXXXX` (where XXXXXX is derived from MAC)
- Default password: Configured in service
- IP: `192.168.4.1`

#### `bool connect_to_wifi(std::string ssid, std::string password)`
Connects to a specific WiFi network with provided credentials

#### `bool disconnect_from_wifi()`
Disconnects from current WiFi network

#### `bool connect_and_fallback(std::string ssid, std::string password)`
Attempts connection with automatic AP fallback:
1. Try to connect to specified network
2. Wait for connection with timeout
3. If fails, automatically switch to AP mode

## Usage Example

### In main.cpp

```cpp
#include "services/WiFiService.h"

WifiService wifi_service = WifiService();

void setup() {
    // Initialize and start WiFi service
    if (!start_service(wifi_service)) {
        logger.error("Failed to start WiFi service");
        return;
    }
    
    // Get network information
    String ip = wifi_service.getIP().c_str();
    String ssid = wifi_service.getSSID().c_str();
    
    logger.info("Connected to: " + ssid);
    logger.info("IP Address: " + ip);
}
```

## Configuration

WiFi configuration is typically stored in `SettingsService` with these parameters:
- `wifi_ssid` - Target network SSID
- `wifi_password` - Network password
- `wifi_hostname` - Device hostname
- `ap_ssid` - Access point SSID (optional)
- `ap_password` - Access point password (optional)

## Integration Points

### Startup Sequence
The WiFiService must be started **before** any network-dependent services:
```cpp

start_service(http_service);        // Then - start web server

```

## Error Handling

The WiFiService implements robust error handling:
- Connection timeouts trigger automatic AP mode
- Failed AP creation is logged as critical error
- WiFi disconnections are detected and can trigger reconnection
- All errors are logged via `RollingLogger`

## Best Practices

1. **Always check WiFi status** before starting network services
2. **Handle both modes** in dependent services (station vs AP)
3. **Don't block on WiFi** operations - use timeouts
4. **Log connection events** for debugging
5. **Test AP fallback** to ensure robot remains accessible

## Troubleshooting

### WiFi Won't Connect
- Verify SSID and password in settings
- Check WiFi signal strength
- Ensure 2.4GHz network (ESP32-S3 doesn't support 5GHz)
- Check for MAC filtering on router

### AP Mode Issues
- Verify no other device is using `192.168.4.1`
- Check AP password meets ESP32 requirements (min 8 characters)
- Ensure power supply is adequate

### IP Address Not Displayed
- WiFi service may not be fully started
- Check serial monitor for connection messages
- Verify UI is refreshing after connection
